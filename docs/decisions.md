* **Proposed**：方向合理，但在实施前仍需负责人明确批准；
* **Deferred**：已有需求，但按触发条件延后；
* **Rejected**：本路线不采用；
* **Superseded**：已被后续决策替代。

## 3. 详细决策

### D-001：文档单一职责和事实来源

状态：Accepted  
日期：2026-07-19

决定：

* PRD 负责产品范围、用户流程、里程碑、验收和非功能要求；
* `docs/architecture.md` 负责数据模型、模块/线程边界、解析/渲染管线、状态不变量和生命周期；
* Codex 清单负责执行阶段、允许文件、自动测试和人工验收；
* 本文负责决策及其理由；
* `联网确认.md` 负责外部事实、证据、原型和实机核查；
* 未来的 `docs/supported-formulas.md` 负责语法、错误码、限制值和验收公式。

理由：在多份文档复制阈值、供应商字段和实现细节，会造成版本漂移并使验收无法判断以谁为准。

后果：引用其他文档时只概述约束并链接到单一事实来源。架构改变先更新 architecture/decisions，再同步 PRD 的产品影响和清单的执行步骤。

### D-002：单图/多图统一 Scene 模型

状态：Accepted  
日期：2026-07-19

决定：所有绘图入口和结果统一为 `PlotSceneRequest` / `PlotSceneResult`；单图是一个 item，多图是多个 item。使用 `PlotItemRequest`、`PlotSceneSpec`、分类型 `PlotItemSpec`、`RenderPlan` 和逐项结果，不保留平行的单图专用请求/结果模型。

理由：从 M0 统一集合模型可以避免 M1.6、OCR 核查和样式/预算再次改造主链路，同时不要求 M0 提前实现多输入 UI。

后果：所有跨线程模型采用不可变快照，item 集合使用 tuple；用户意图模型不暴露采样网格等实现参数。

### D-003：唯一常驻 RenderActor

状态：Accepted  
日期：2026-07-19

决定：M1 起所有完整绘图请求进入一个常驻、单线程串行消费的 RenderActor。取消 GUI 主线程显函数旁路和“复杂时才启用后台线程”的双路径。

理由：解析、NumPy、字体与 Matplotlib 的耗时会随输入和机器变化；运行时猜测“简单”会产生不可测试的 UI 阻塞边界。Matplotlib 不是线程安全组件，单消费者比多个渲染 worker 争抢全局锁更容易证明正确。

替代方案：

* GUI 主线程同步绘制：Rejected，会冻结界面且难以统一取消/关闭语义；
* 每个请求一个 Worker + 全局锁：Rejected，线程生命周期和队列积压复杂，锁并不解决过期信号；
* 多进程：Rejected，超出首版范围和打包/内存预算。

后果：Actor 只保留一个运行任务和一个 latest-wins 待任务；request_id、scene revision 和协作取消共同防止旧结果覆盖。

### D-004：TaskPhase + scene revision 状态模型

状态：Accepted  
日期：2026-07-19

决定：`TaskPhase` 只表示当前唯一的用户可见前台活动，不含就绪或错误通知。首版 AppController 不同时调度截图/OCR 与绘图任务；RenderActor 与 RecognitionWorker 可以同时常驻，但冲突操作在另一前台任务活动时必须被禁用或明确拒绝。保存当前 revision、当前请求编号、最后成功结果、结果 revision、错误通知和任务上下文；`has_plot_result`、`copy_enabled`、`result_is_stale`、`is_ready` 全部派生。

理由：把阶段、结果存在性、dirty 和错误塞入同一枚举或重复布尔字段，会形成不可达组合和失配状态。

后果：任何影响结果的编辑立即递增 revision；revision 不防抖。旧图允许复制，但 UI 必须显示其对应旧输入。若未来要求 OCR 与绘图真正并发，必须以新决策改为可表达正交活动的状态模型，不能给六值 `TaskPhase` 增加含糊组合值。

### D-005：自有受限 AST 作为首选解析安全边界

状态：Accepted（修改后采纳）  
日期：2026-07-19

原提议：受限 parser 可以直接构造内部 AST 或受限 SymPy 节点。

修改后的决定：优先先构造项目自有受限 AST，在转换到 SymPy/数值函数前完成字符、token、节点、深度、数字和指数限制。只有原型证明直接构造受限 SymPy 节点不会在验证前触发高成本求值时，才允许该实现。

理由：仅使用白名单名称并不能阻止巨大整数、深层幂或构造期自动求值消耗资源。安全边界必须先于可能执行代数计算的库调用。

后果：手动文本与 OCR LaTeX Adapter 最终进入同一 tokenizer、AST 和结构验证器；禁止对原始输入直接使用 eval/exec、无约束 sympify、通用 parse_expr 或宽松 LaTeX parser。

### D-006：一般非退化直线加入 M1.5

状态：Accepted  
日期：2026-07-19

决定：M1.5 产品名调整为“受限直线、圆与圆锥曲线方程绘图”，支持 `x=2`、`x+y=1`、`2x-y+3=0` 等数值系数非退化一般直线，分类为 `LINE_EQUATION`，规范为 `LineSpec(Ax+By+C=0)`。

理由：现有教学场景已包含直线与圆锥曲线位置关系；只支持 `y=kx+b` 会错误拒绝最常见的竖直直线，且一次方程分类比圆锥曲线分类简单、边界清晰。

不包含：方程组、自动求交、参数直线、不等式或符号参数。

后果：直线与视口矩形求交采样，不使用 contour。若后续产品范围评审推翻本决策，必须新增替代决策，说明 `x=2` 的明确“不支持”错误和教学影响，不能把它误报为圆锥曲线错误。

### D-007：圆锥曲线主实现采用参数化采样

状态：Accepted（待原型校准）  
日期：2026-07-19

决定：受限轴向平行、非退化圆/圆锥曲线在分类得到几何 Spec 后，使用参数化采样。二维 contour 只保留为诊断对照或未来任意隐式曲线的独立研究，不是当前主实现。

理由：已知曲线类型时，参数化采样能明确分支、减少二维网格内存、提供精确几何元数据，并让资源预算以点数和分支数表达。contour 的空路径、网格分辨率和裁切判定不应成为受限几何模型的核心正确性基础。

风险：双曲线/抛物线必须可靠求出与视口相关的有限参数区间，极端系数与裁切仍需原型和教材样例验证。

后果：删除二维网格尺寸/总单元数作为主模型字段和发布预算；改用单项/场景采样点、分支、内存、耗时目标和输出尺寸预算。

### D-008：缓存延后到基准证明需要之后

状态：Deferred  
日期：2026-07-19

决定：缓存不是 M1、M1.5 或 M1.6 发布门槛。M1.6 先建立 item fingerprint、逐阶段耗时、可扩展命中字段和可复现基准；只有明显瓶颈证据才进入 M1.6.1。

理由：解析和采样缓存会引入版本化、视口依赖、内存所有权和失效正确性。没有基准时成本大于收益，并可能让旧结果污染当前场景。

后果：若实施，先复用上一成功场景的已验证 Spec，再评估采样结果，最后才考虑有界 LRU；禁止无界缓存和过窄缓存键。

### D-009：单一类型检查器拟采用 Pyright

状态：Proposed  
日期：2026-07-19

决定：原则上只选择一个类型检查器，建议 Pyright；Models/Engine 严格，UI/PySide6 信号层先标准。pytest 与类型检查分开报告，不同时强制 mypy。

待批准：安装和配置 Pyright、修改 `pyproject.toml`/相关配置以及锁文件。当前文档评审不安装、不配置。

批准条件：阶段 5 开始时确认工具、版本、运行命令、忽略策略和对现有代码的迁移范围。

### D-010：CI 与扫描作为独立后期阶段

状态：Proposed  
日期：2026-07-19

决定：先建立统一的本地质量检查命令；CI、依赖/漏洞扫描和敏感信息扫描在阶段 29 单独实施。Windows UI、截图、剪贴板、多显示器和打包以 Windows runner/实机为准。

待批准：创建或修改 `.github/workflows`，以及安装任何 CI、漏洞或敏感信息扫描工具。当前文档评审不创建 CI。

明确不要求：Ubuntu 完整运行 Windows GUI 验收、Linux Docker 构建正式 Windows 安装包、两次打包产物二进制哈希完全一致。

### D-011：OCR 结果与输入分析结果分离

状态：Accepted  
日期：2026-07-19

决定：供应商转换只产生 `OcrResult`；本地安全分析产生 `InputAnalysisResult`；核查 UI 用 `FormulaReviewModel` 组合原图与两类结果。SimpleTexService 不负责数学规范化、分类或绘图判断。

理由：供应商契约是易变外部边界，数学解释是本项目安全边界，合并模型会让网络服务绕过本地验证并使测试职责不清。

### D-012：QSettings 不存真实密钥

状态：Accepted  
日期：2026-07-19

决定：SettingsService 只保存普通配置或凭据引用，真实密钥由独立 `CredentialStore` 接口管理。具体存储后端在 M3 外部核查和产品批准后确定。

理由：QSettings 不是密钥库；桌面发布包也不能内嵌供应商 APP Secret 或共享核心凭据。

### D-013：可访问性基线前移到静态 UI

状态：Accepted  
日期：2026-07-19

决定：键盘操作、Tab 顺序、焦点可见、非颜色状态、触控目标和高 DPI 文字布局在静态 UI 阶段完成；后期样式阶段只精修并做最终验收。

理由：这些约束影响控件结构、布局和状态表达，后补会导致大面积返工，且不属于装饰性样式。

### D-014：Windows 11 主支持、Windows 10 条件兼容

状态：Accepted（兼容矩阵待实机验证）
日期：2026-07-19

决定：Windows 11 64 位是主要支持平台；Windows 10 1809 及以上 64 位是面向课堂存量设备的条件兼容目标，不与 Windows 11 构成同等级、无期限的支持承诺。M0 固定依赖组合时必须保留该兼容目标或形成明确降级记录；M6 发布前必须用实机矩阵确认最终最低版本、已知风险和维护终点。

理由：课堂仍可能存在 Windows 10 设备，但其常规支持状态与 Qt 技术兼容范围不能混为一谈。把“主要支持”和“兼容目标”分开，既保留现实覆盖面，也避免对已结束常规支持的平台作无证据承诺。

替代方案：当前不采用“只支持 Windows 11”；也不采用“Windows 10 与 Windows 11 等级相同且长期支持”。若依赖或实机证据不能满足兼容目标，须以新决策收缩平台范围并同步 PRD、联网确认和发布验收。

后果：阶段 0 记录可复现的 Windows/Python/Qt 组合和兼容风险；阶段 28/30 完成打包、安装、升级、卸载和目标设备矩阵。未通过的 Windows 10 版本不得继续出现在正式支持声明中。

### D-015：P0-02 课堂基准设备延期实机验证

状态：Accepted  
日期：2026-07-20  
负责人：项目所有者（批准延期实机验证）  
关联需求/核查项：P0-02

背景：当前无法获得真实课堂设备。当前高配置 Windows 11 电脑仅是开发机和开发参考环境，不代表学校电脑、教学一体机或经确认的代表性低配置设备。

决定：P0-02 以项目所有者批准的延期实机验证方式关闭；当前阶段不因缺少真实课堂设备而阻塞开发。

理由：开发不应因暂时缺少真实课堂设备而无期限停滞，但开发机结果不能替代课堂设备结论。

替代方案：不采用把当前开发机结果宣称为课堂设备验证通过；也不将本延期关闭视为课堂兼容性或性能达标结论。

后果与风险：当前设备上的性能、DPI、触控和多显示器观察均不得外推。阶段 12、15 等较早性能基准若只能在开发机执行，只能标记为“开发参考数据”。在后续实机验证完成前，不得声称软件已满足课堂设备兼容性或性能要求。

复查或恢复条件：阶段 26 开始时重新打开 P0-02 的课堂设备实机验证风险，并在真实学校设备或项目所有者批准的代表性低配置 Windows 10/11 设备上验证触控、100%–200% DPI、多显示器、界面可用性以及典型启动和绘图性能。

最终门禁：上述实机验证最迟在阶段 30 发布候选验收前完成，或形成新的明确发布降级决定；否则不得宣称课堂设备兼容或形成公开发布候选。阶段 26 和阶段 30 已分别承担课堂 DPI、触控、多显示器及最终实机发布验收，本延期事项不得永久搁置。
