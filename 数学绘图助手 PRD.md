## 1. 产品概述

“数学绘图助手”是一款面向教师电脑和教学一体机的 Windows 桌面软件。

软件帮助教师将手动输入或截图识别得到的数学公式、方程快速转换为清晰的函数或平面曲线图像，并复制到 Windows 剪贴板，以便粘贴到希沃白板、PowerPoint、Word、WPS、教案或其他教学软件中。

核心 MVP 重点解决：

1. 公式输入不方便；
2. 临时绘制函数图像耗时；
3. 教学软件之间转移函数图像步骤较多；
4. 手写公式识别结果需要人工确认；
5. 教学场景需要操作简单、反馈明确、触控友好。

产品定位边界：

* 核心价值是“快速生成可直接用于课堂材料的静态数学图像”；
* 不是通用计算机代数系统，不提供任意方程求解或证明；
* 不是动态几何软件，不在首期替代 GeoGebra 等交互式工具；
* 手动输入、绘图、预览和复制必须可离线工作，OCR 是可选在线能力；
* 对已声明支持的公式优先保证数学正确性、视觉正确性和可解释错误，不以“尽量画出来”替代验证。

## 2. 产品目标

### 2.1 核心目标

用户可以完成以下流程：

```text
输入公式或方程
→ 检查和确认
→ 识别绘图类型并解析
→ 生成函数或平面曲线图像
→ 界面预览
→ 复制到剪贴板
→ 粘贴到其他教学软件
```

后续支持：

```text
Windows 截图或导入图片
→ SimpleTex 识别
→ 用户核查和修改
→ 解析公式或方程并识别绘图类型
→ 生成图像
→ 复制到剪贴板
```

### 2.2 产品成功标准

核心 MVP 及其后的 M1.6 教学场景增强按对应里程碑应满足：

* 普通教师无需理解 Python、SymPy 或 Matplotlib；
* 手动输入后，从提交到得到图像只需一次主要操作；
* 输入错误时给出可理解的中文提示；
* 不因错误公式导致程序崩溃；
* M1 支持已声明的单自变量显函数，M1.5 支持已声明的一般直线、圆与圆锥曲线方程，M1.6 支持把多个已支持图像叠加到同一坐标系；
* 已声明支持的验收公式在自动测试矩阵中必须全部通过；
* 无网络时仍可完成手动输入、绘图、预览和复制；
* 生成的图片可以复制并粘贴到验收矩阵规定的 Windows 教学软件；
* 界面在 100%、125%、150%、175% 和 200% 缩放下可用；
* 不将 API 密钥、课堂截图或学生信息写入不安全日志；
* 核心数学功能可以通过自动化测试验证；
* 性能、触控尺寸、启动耗时和支持软件版本采用第 7 节及 `联网确认.md` 中的可测口径。

### 2.3 交付定义与优先级

为避免“首版”“第一阶段”和“发布版”含义混乱，统一定义：

* **核心 MVP**：M0、M1、M1.5，完成离线手动绘图与复制；
* **教学场景增强**：M1.6，在 M1 与 M1.5 验证完成后实现同一坐标系多图像；优先于 OCR 开发，但优先级为 SHOULD，不阻塞核心 MVP 验收；
* **OCR Beta**：M2、M3、M4，完成本地图片、SimpleTex 和 Windows 截图识别；
* **1.0 发布候选**：M5、M6 完成后形成；
* **MUST**：对应里程碑的发布阻塞需求；
* **SHOULD**：重要但可在有明确降级说明时延期；
* **COULD**：候选增强，不计入对应里程碑完成标准；
* **OUT**：当前路线明确不做。

## 3. 目标用户

### 3.1 主要用户

核心 MVP 的主要用户：

* 数学教师；
* 教研人员；
* 使用教学一体机授课的教师。

信息技术教师、科学教师和学生是后续扩展用户；在数据绘图、多变量模型等能力尚未纳入前，不作为核心 MVP 的主要验收对象。

### 3.2 用户特征

主要用户可能：

* 不熟悉 LaTeX；
* 不熟悉编程；
* 使用鼠标、触控屏或教学一体机；
* 需要在课堂上快速完成操作；
* 更关注结果是否正确、清晰和易复制；
* 无法处理复杂报错信息。

### 3.3 核心使用场景

1. 教师输入 `y=sin(x)`，快速生成并复制函数图像；
2. 教师输入椭圆、双曲线或抛物线方程，得到完整且比例正确的曲线；
3. 教师把函数、直线与圆或圆锥曲线叠加在同一坐标系，用于比较图像、讲解位置关系和展示综合题条件；
4. 教师导入或截取纸面公式，核对软件实际理解的表达式后绘图；
5. 教师在课堂中修改公式或视口，重新生成并复制图片；
6. 网络不可用时，教师仍能使用全部本地绘图能力。

## 4. 产品范围

### 4.1 M1：单自变量显函数

第一阶段只支持单自变量显函数：

```text
y = f(x)
```

典型输入：

```text
y = x²
y = x^2
y = x**2
y = sin(x)
y = 1/x
y = sqrt(x)
```

M1 优先支持高中及以下常见函数：

* 一次函数；
* 二次函数；
* 反比例函数；
* 幂函数；
* 指数函数；
* 对数函数；
* 绝对值函数；
* 基本三角函数；
* 简单复合函数。

M1 只允许显函数右侧出现自变量 `x`。`y=x²` 等纵向抛物线属于显函数支持范围，但这不等于已经支持完整圆锥曲线。

### 4.2 M1.5：受限直线、圆与圆锥曲线方程

v0.1 未覆盖完整圆锥曲线：它限定 `y=f(x)`，且排除了隐式方程，因此不能完整绘制圆、椭圆、一般双曲线和侧开口抛物线。

M1.5 增加单条、实数、非退化的一般直线，以及坐标轴平行的二次曲线方程：

* 一般直线，包括竖直直线；
* 圆（在课程内容和产品入口中单列为“圆与方程”）；
* 椭圆；
* 双曲线；
* 抛物线；
* 标准式、平移后的标准式，以及与之等价的无旋转展开式；
* 方程左右两侧均可包含表达式，软件统一化为 `F(x, y) = 0`；
* 参数必须已经代入有限实数，不接受未赋值的 `a`、`b`、`p` 等符号参数。

必须支持的输入示例：

```text
x = 2
x + y = 1
2*x - y + 3 = 0
x^2 + y^2 = 25
(x-2)^2 + (y+1)^2 = 9
x^2/9 + y^2/4 = 1
(x-1)^2/9 + (y+2)^2/4 = 1
x^2/9 - y^2/4 = 1
y^2 = 8*x
x^2 = -4*y
(y-1)^2 = 4*(x+2)
x²/9 + y²/4 = 1
```

M1.5 明确不支持：

* 含 `xy` 项的旋转圆锥曲线；
* 退化为点、两条直线或空集的二次方程；一次方程只通过明确的直线分类进入支持范围；
* 无实数图形的方程；
* 任意非二次隐式方程；
* 未赋值的符号参数；
* 在单项输入中包含多条方程、方程组、不等式和参数方程；M1.6 只允许把多条受支持图像分别放入独立输入行；
* 自动标注焦点、准线、顶点、离心率或渐近线；
* 参数滑块和实时联动。

最后两项列入课堂体验阶段的候选增强，不阻塞 M1.5 的静态图像闭环。

### 4.3 M1.6：同一坐标系多图像

M1.6 面向高中教学中的函数比较、函数与圆锥曲线位置关系、参数变化对比和解析几何综合题，在一张图片的同一坐标系内叠加多个当前已支持的图像。

M1.6 首期范围：

* 每个图像使用独立输入行，可添加、删除和调整顺序；不依赖逗号、分号或换行猜测多公式边界；
* 每个输入项可独立为 M1 已支持的显函数，或 M1.5 已支持的一般直线、圆与圆锥曲线方程；
* 允许显函数与圆/圆锥曲线混合，例如 `y=x+1` 与 `x^2+y^2=25`；
* 所有输入项共享一个 Viewport、坐标轴、网格、图片尺寸和输出图片；
* 场景中只要包含圆或圆锥曲线，默认对整个坐标系使用单位等比例，用户可明确修改；
* 按输入顺序自动分配稳定、易区分且兼顾色觉差异的颜色与线型，并默认显示图例；同一输入顺序重新生成时样式保持稳定；
* 图例使用用户输入的简化文本或安全生成的显示标签，过长时截断但可查看完整内容；
* 每项保留稳定 `item_id`，解析、警告和错误必须定位到对应输入行；
* 首期采用原子生成：任一输入项无效时，本次场景整体生成失败并保留上一张成功图片，不得静默漏画或把部分结果展示为完整成功；
* 首期每个场景受配置的项目数上限约束；同时设置单项采样点、场景总采样点、分支数、内存、渲染耗时目标和输出图片大小上限。各限制值在 M1.6 开始前依据 M1、M1.5 基准测试冻结到配置和 `docs/supported-formulas.md`，本 PRD 不复制具体数值；
* 首期只负责叠加绘图，不自动求交点、切线、公共区域、方程组解或图像之间的数学关系；
* OCR 首期仍按一次识别结果填充一个输入项，不承诺从一张图片自动拆分多条公式。

建议的高中教学验收场景：

```text
y=x 与 y=2*x+1                         # 一次函数比较
y=x^2 与 y=2*x+1                       # 函数图像与交点观察
y=sin(x) 与 y=cos(x)                   # 三角函数比较
y=x+1 与 x^2+y^2=25                    # 直线与圆的位置关系
y=x 与 x^2/9+y^2/4=1                   # 直线与椭圆综合题
y=2 与 y^2=8*x                         # 直线与抛物线位置关系
```

### 4.4 当前路线暂不支持

核心功能稳定前，不支持：

* 除第 4.2 节受限一般直线、圆与圆锥曲线外的任意隐式方程；
* 参数方程；
* 极坐标方程；
* 分段函数及其可视化编辑器；
* 不等式区域；
* 方程组自动联立求解；
* 微分方程；
* 三维图；
* 几何作图；
* 动态几何；
* 数据表格绘图；
* MathLive；
* Qt WebEngine；
* QML 或 Qt Quick；
* 自研全屏截图遮罩；
* 数据库；
* 插件系统；
* 自动更新；
* 云端账号体系；
* 云端遥测和用户行为分析；
* 网络服务器；
* 多进程架构。

### 4.5 圆锥曲线与多图像实施顺序决策

采用“现在纳入 PRD 和架构，先验证单图能力，再实现多图场景”的方案：

1. M1 先验证安全解析、一维采样、渲染、预览和复制的最小闭环；
2. M1.5 再引入二变量方程分类、一般直线和参数化圆锥曲线采样、等比例视口与完整分支绘制，避免拖大 M1；
3. M1.5 必须早于 OCR，防止公式核查界面、数据模型和解析管线固化为只有 `y=f(x)`；
4. M0 即预留 `Scene + items[]` 集合模型；M1 和 M1.5 分别以单项场景验证，不要求提前完成多输入 UI；
5. M1.6 在 M1 与 M1.5 验收通过后实现，默认排在 OCR 前，避免 OCR 核查界面再次从单输入改造成多输入；
6. M1.6 为 SHOULD；如果确需延期，必须记录降级决定，但不得撤销 M0 的集合模型预留；
7. OCR 只是输入方式，不应决定数学引擎支持范围；
8. M1.5 或 M1.6 失败时不影响已完成的单显函数能力，可独立回退和验收。

## 5. 核心用户流程

### 5.1 手动输入绘图

```text
用户输入公式或方程
→ 设置或使用默认视口
→ 点击“生成图像”
→ 软件规范化并识别绘图类型
→ 软件验证并生成图像
→ 显示预览
→ 用户点击“复制图片”
→ 图片写入 Windows 剪贴板
```

### 5.2 直线、圆与圆锥曲线方程绘图

```text
用户输入含 x、y 的单个一次或二次方程
→ 软件将方程规范化为 F(x, y) = 0
→ 软件验证变量、次数、系数、退化与实图形条件
→ 软件识别为直线、圆、椭圆、双曲线或抛物线
→ 使用或自动建议 x、y 视口
→ 以明确的几何参数化策略生成完整可见曲线
→ 显示图形类型、规范化方程和必要警告
→ 用户复制图片
```

如果曲线在当前视口内完全不可见，本次生成失败，界面保留用户输入和上一张成功图片并提示调整范围，不得把空白图片误报为成功曲线。部分裁切仍可成功，但必须附带警告。

### 5.3 同一坐标系多图像

```text
用户添加多个独立公式/方程输入行
→ 设置或使用共享视口
→ 点击“生成图像”
→ 软件逐项规范化、分类和验证
→ 任一项失败时定位到对应输入行并停止本次生成
→ 全部有效时分别采样并在同一 Figure 中叠加渲染
→ 显示稳定颜色、线型、图例和逐项警告
→ 用户复制一张合成图片
```

### 5.4 导入图片识别

```text
用户选择本地图片
→ 软件显示图片预览
→ 用户点击“识别公式”
→ SimpleTex 返回识别结果
→ 软件进入公式核查阶段
→ 同时显示原图、OCR 文本、软件实际理解的规范化表达式和绘图类型
→ 用户修改或确认
→ 软件生成图像
```

### 5.5 Windows 截图识别

```text
用户点击“截图识别”
→ 软件调用 Windows 截图工具
→ 用户框选公式
→ 截图进入剪贴板
→ 软件判断这是本次外部截图
→ 软件进入 OCR
→ 用户核查原图、识别文本和软件实际理解的表达式
→ 软件生成图像
```

### 5.6 错误恢复

```text
发生错误
→ 界面显示简洁中文说明
→ 保留用户原始输入
→ 允许用户修改后再次提交
→ 不要求重启程序
```

如果错误发生前已经有一张成功图像，旧图仍可预览和复制，但必须显示“输入已修改”或“本次生成失败”，避免用户误把旧图当作当前输入结果。

## 6. 功能需求

### 6.1 公式与方程输入

软件应提供：

* M1、M1.5 提供单个公式/方程输入；M1.6 提供多个相互独立的输入行；
* 根据当前能力展示显函数和圆锥曲线示例；
* 生成按钮；
* 清空按钮；
* 最近一次有效输入保留；
* 支持键盘回车提交；
* 输入错误后不清空原内容；
* OCR 结果可直接进入当前选中的输入行修改；
* 显示软件识别出的绘图类型；
* 允许用户查看规范化后的表达式，但不要求用户理解 LaTeX；
* 输入改变后标记现有图像为“对应旧输入”。

核心 MVP 不提供完整 LaTeX 编辑器。

#### 6.1.1 输入语义

M1 必须建立并测试明确的输入契约：

* `^` 和 `**` 均表示幂；支持常见 Unicode 上标 `²`、`³`；
* 支持受控的隐式乘法，例如 `2x`、`2(x+1)` 和 `(x+1)(x-1)`；
* `-x^2` 按 `-(x^2)` 解释；
* `abs(x)` 和 `|x|` 表示绝对值；
* `ln(x)` 表示自然对数，`lg(x)` 表示常用对数，指定底数使用 `log(x, b)`；裸的 `log(x)` 在解析阶段直接拒绝，并提示改用 `lg(x)`、`ln(x)` 或 `log(x, b)`；
* 三角函数默认使用弧度，角度制列为后续能力；
* 手动文本和 OCR LaTeX 可以有不同的前置规范化步骤，但必须进入同一结构验证器；
* 未知符号、歧义写法或未完整解析的尾部文本必须拒绝，不允许静默截断。

M1 同步建立 `docs/supported-formulas.md`，作为输入语法、示例、错误码和自动测试的单一事实来源。

---

### 6.2 绘图参数与视口

核心 MVP 至少提供：

* x 轴最小值；
* x 轴最大值；
* y 轴最小值和最大值：显函数可自动，圆与圆锥曲线必须有有效值；
* 自动视口与手动视口切换，用户手动值优先；
* 是否显示网格；
* 图片宽度和高度的默认值；
* 坐标比例模式：显函数默认自动，圆与圆锥曲线默认单位等比例；
* 自动使用有上限的合理采样点和分支数量。

可以后续增加：

* 线宽；
* 曲线颜色；
* 坐标轴样式；
* 背景透明；
* 标题；
* 用户自定义颜色、线型和图例标签。

核心 MVP 必须对坐标范围、极小跨度、图片尺寸、DPI、单项/场景总采样点、分支数和内存设置上限与下限，避免卡顿或内存异常。具体阈值进入配置、测试和 `docs/supported-formulas.md`，不得只存在于 UI 控件中。

绘图视觉规则：

* 坐标轴在视口包含原点时穿过原点，否则位于图框边缘并保持刻度可读；
* 默认显示 `x`、`y` 轴标签；
* 圆不得因图片宽高或窗口缩放被拉伸成椭圆；
* 图像边距必须保证刻度和轴标签不被裁切；
* 自动视口无法可靠判断时回退为 `x,y ∈ [-10,10]` 并允许用户调整；
* 用户选择手动视口后，不得在重新绘图时被自动范围静默覆盖。

---

### 6.3 图片导入与公式识别

SimpleTex 接入后应满足：

* 支持本地图片；
* 支持 Windows 截图；
* OCR 请求不阻塞界面；
* 设置明确的连接和读取超时；
* 支持取消，网络失败、鉴权失败、配额不足和限流后可以恢复；
* 不自动信任识别结果；
* 用户必须同时看到原图、识别文本、规范化表达式和识别出的绘图类型；
* 不将识别结果直接无提示地送入绘图；
* 即使当前 API 返回置信度，也不把该字段作为永久必需契约；
* API 返回结构以接入时的官方文档、固定版本和契约测试为准；
* 只有用户主动选择图片或发起截图识别后才允许上传，不得监听并上传任意剪贴板图片；
* 无网络时明确告知 OCR 不可用，但不得影响本地绘图；
* 重试必须有次数上限，并避免一次用户操作造成重复计费。

导入图片在预览或上传前必须验证：

* 扩展名与实际解码格式；
* 文件编码大小；
* 解码后的宽、高和总像素数；
* 是否可完整解码；
* 是否可能造成解压炸弹或异常内存占用；
* 临时字节和临时文件在任务结束、取消或应用关闭后的清理策略。

---

### 6.4 直线、圆与圆锥曲线绘图

M1.5 必须：

* 接受恰有一个等号的方程，安全解析左右两侧并构造 `F(x, y) = 0`；
* 将有限小数尽可能转换为精确有理数后再分类，避免近零浮点误判；
* 验证自由变量只包含 `x`、`y`，且所有系数为有限实数；
* 验证分母不含 `x`、`y`，受限展开后为总次数为 1 或 2 的多项式；
* 将非退化一次方程分类为一般直线，并明确支持竖直直线；
* 规范化为 `Ax² + Bxy + Cy² + Dx + Ey + F₀ = 0`，并在 M1.5 要求 `B = 0`；
* 结合判别式与退化判定识别圆、椭圆、双曲线或抛物线；
* 明确拒绝退化曲线、无实点曲线和零方程；
* 直线与视口矩形求交；圆、椭圆、双曲线和抛物线使用各自参数化采样，不把方程强行求解为若干 `y=f(x)` 分支；
* 对方程整体乘非零常数或左右交换保持相同分类和图形；
* 默认单位等比例绘制，正确保留闭合曲线、双曲线多分支和抛物线开口方向；
* 根据几何曲线与视口的相交/裁切结果判断当前视口内是否存在曲线；
* 当前视口完全无曲线时使本项失败并提示调整范围；曲线部分裁切或采样精度不足时允许成功，但必须给出可理解警告；
* 限制单项/场景总采样点、分支数、内存和输出尺寸，并及时释放数组和 Figure。

M1.5 不自动求解焦点、准线、顶点、离心率、渐近线或交点；这些教学标注在有独立需求和验收标准后进入后续里程碑。

---

### 6.5 同一坐标系多图像

M1.6 必须：

* 提供添加、删除和重排输入项的明确操作，并至少支持键盘完成；
* 每个输入行独立执行现有的规范化、解析、分类和验证管线，不增加用于拆分多公式的新语法；
* 使用一个 `PlotSceneRequest` 表达共享视口与 `items[]`，单图能力是 `items` 仅含一项的特例；
* 场景级验证项目数、总采样点、总分支数、预计内存和图片尺寸；
* 在采样或渲染前完成所有输入项验证；任一项失败时返回带 `item_id`、输入行和源位置的 ErrorInfo；
* 在一个 Figure 和一套坐标轴上渲染全部有效项，图例顺序与输入顺序一致；
* 同一个 `item_id` 在增删其他项或重新生成时尽量保持颜色和线型稳定；
* 对色觉差异不只依赖颜色区分，打印或灰度显示时仍能通过线型或图例识别；
* 逐项返回规范化表达式、绘图类型、可见片段数和警告，并返回场景级成功、失败和资源信息；
* 如果某一有效图像在当前视口不可见，按原子生成规则使本次场景整体失败；首版不提供“确认后接受部分可见结果”的旁路，未来若启用该行为必须先新增决策并修订架构不变量；
* 复制、预览和过期状态以整张场景图片为单位，不允许只复制未明确标识的部分结果。

M1.6 不要求：

* 自动计算或标注交点；
* 联立求解输入项；
* 自动填充阴影、公共区域或不等式区域；
* 为每项配置独立视口或独立坐标轴；
* 从一段混合文本或一张截图自动拆分多个公式。

---

### 6.6 图像预览

预览区域应：

* 在无图片时显示占位状态；
* 在生成期间显示处理中状态；
* 显示最新有效绘图结果；
* 保持图片比例；
* 不因窗口缩放导致图片严重变形；
* 生成失败时保留上一张成功图片，除非用户主动清除；
* 支持重新复制当前图片；
* 显示当前图片对应的全部原始输入、逐项规范化表达式和绘图类型；
* 输入已改变、任务失败或结果过期时，明确标记图片不是当前输入的新结果；
* 空视口警告不得伪装成成功绘制。

---

### 6.7 剪贴板

软件应支持：

* 将当前绘图结果写入 Windows 剪贴板；
* 判断当前是否存在可复制图片；
* 复制成功后显示明确提示；
* 输出格式必须通过 PowerPoint、Word、WPS、希沃白板或已确认替代软件的粘贴验收矩阵；
* 软件自己写入剪贴板时，不触发截图 OCR；
* 外部截图写入和内部绘图写入必须区分；
* 不仅依靠一次 `dataChanged` 信号就直接开始 OCR。

---

### 6.8 状态反馈

界面应显示：

* 当前阶段；
* 正在执行的任务；
* 成功提示；
* 可恢复错误；
* 网络超时；
* 截图取消或超时；
* 复制成功；
* 当前公式或方程不受支持；
* 识别出的绘图类型；
* 多图场景中出错或不可见项目对应的输入行；
* 当某个显函数的受预算采样序列触发已定义的密集振荡代理指标时，显示“当前区间内可能包含密集振荡，请确认是否需要调整范围”等警告，不得静默报告成功；代理指标、阈值和验收样例在阶段 8 写入 `docs/supported-formulas.md`，不把任意函数的“周期数”作为未经定义的实现口径；
* 曲线不在当前视口、结果对应旧输入或当前结果已过期。

按钮状态应与当前阶段一致，例如：

* OCR 进行中时避免重复提交；
* 绘图进行中时避免同一请求重复启动；
* 首版任一截图、OCR 或绘图前台任务进行中时，禁用或明确拒绝会启动另一类前台任务的冲突操作；
* 没有图片时禁用复制按钮；
* 关闭程序时不再接受新任务。

---

## 7. 非功能需求

### 7.1 易用性

* 默认界面使用中文；
* 关键按钮文字清晰；
* 不向普通用户显示 Python 异常堆栈；
* 主要触控目标建议不小于 44×44 逻辑像素，最终以目标一体机实测为准；
* 手动输入绘图从输入完成到预览只需一次主要操作，OCR 流程不超过五次主要操作；
* 默认参数应能直接生成大多数已支持高中函数及标准圆锥曲线图像；
* 所有核心操作可以用键盘完成，Tab 顺序合理且焦点清晰可见；
* 错误、警告和成功状态不得只用颜色区分；
* 在系统字号和缩放增大时，文字不得被截断到无法操作；
* OCR 核查页必须让不熟悉 LaTeX 的用户看懂软件将要绘制什么。

### 7.2 性能

M0 结束前选定一台可复现的课堂基准设备；以下为待原型校准的目标，调整必须形成决策记录：

* 冷启动到可输入状态 P95 不超过 5 秒；
* M1 常见显函数从提交到预览 P95 不超过 1 秒；
* M1.5 默认视口和默认参数化采样从提交到预览 P95 不超过 2 秒；
* M1.6 在项目数与总采样量默认上限内，从提交到多图场景预览 P95 目标不超过 3 秒，具体上限由原型校准；
* 用户交互期间 GUI 主线程单次不可响应时间不超过 200 毫秒；
* GUI 主线程不得执行网络等待、图片大规模解码、完整 Engine、采样或 Matplotlib 渲染；
* 所有完整绘图请求必须进入唯一常驻 RenderActor；
* Matplotlib 访问必须在 RenderActor 中串行化并使用非交互式 Agg 后端，不得并发修改 Artist；
* 任务结果必须携带请求编号；
* 旧任务结果不得覆盖新任务结果；
* 单项及场景总采样点数、分支数、项目数、单次内存峰值和输出图片大小必须有硬上限并纳入边界测试。

首次使用 P95 判断里程碑是否达标前，必须先在基准记录中冻结测量协议：样本量与预热次数、冷/热启动区分、计时起止点、成功预览的判定、异常样本处理、百分位计算方法、重复运行规则以及完整软硬件环境。阶段 12 建立 M1 协议，阶段 15 沿用或以有记录的理由修订；本轮不预填尚待基准验证的数值。

### 7.3 稳定性

* 不合法输入不得导致程序崩溃；
* 网络错误不得导致程序无法继续操作；
* 应用关闭时必须正确停止、等待或放弃后台任务；
* 不出现 `QThread: Destroyed while thread is still running`；
* Worker 不直接修改 QWidget；
* 所有 QWidget 更新在 GUI 主线程完成；
* 取消采用协作式取消或结果作废，不强制终止正在执行的 Python 代码；
* 绘图或 OCR 失败时，上一张成功图片和复制能力按状态规则保留；
* 无可见曲线不得作为无警告的成功结果；
* 解析器、图片解码和状态转换必须包含随机或边界输入“不崩溃”测试。

### 7.4 隐私

* 默认不保存课堂截图；
* 核心 MVP 默认不发送遥测、使用统计或崩溃报告；
* 日志不记录完整截图内容；
* 日志不记录完整 API 密钥；
* OCR 前必须说明供应商、上传内容、处理目的和是否保存，并由用户主动确认；
* 涉及学生姓名、成绩或其他个人信息的截图应提示用户先裁剪或遮挡；
* 不满十四周岁未成年人的个人信息按敏感个人信息风险处理，真实 OCR 接入前完成合规评审；
* 原图、临时文件和内存副本只保留完成当前任务所需的最短时间；
* 用户可以查看日志位置、清理本地历史和撤销已保存凭据；
* SimpleTex 的图片保存期限、删除机制、处理地点、未成年人和教育场景条款必须在 M3 前核实；
* 未完成上述核实时，OCR 只能用于开发测试，不能进入 1.0 发布候选。

### 7.5 可维护性

* UI 和数学引擎分离；
* Engine 不依赖 PySide6；
* Service 不决定数学绘制规则；
* Worker 不承担界面设计；
* AppController 负责任务协调；
* 核心 Engine 可以使用 pytest 独立测试；
* 显函数与圆锥曲线使用带类型的 item 内部模型，单图和多图统一使用 `Scene + items[]`，不依靠散落的布尔标记判断；
* 输入契约、错误码和验收样例由文档与参数化测试共同维护；
* 固定并记录经过验证的 Python 与依赖版本，`uv.lock` 纳入版本控制，依赖变更同步更新锁文件；
* 公开函数、方法和 dataclass 字段使用明确类型；跨线程信号传明确结果对象，不传无结构 dict；
* 只选择一个强制类型检查器，Models/Engine 严格，UI/PySide6 边界可分阶段收紧；类型检查与 pytest 分开报告；
* 发布构建必须可复现到依赖与构建步骤，不要求二进制逐字节完全一致。

### 7.6 安全性

* 禁止使用 `eval`、`exec`、无约束 `sympify` 或等价的可执行式解析；
* 规范化不构成安全边界，解析后必须检查表达式树、符号、函数、运算和复杂度；
* 图片必须先通过类型、大小、像素数和解码验证再进入 OCR；
* 网络请求必须验证 TLS 证书，限制重试、响应体大小和并发；
* 不在桌面客户端内嵌供应商 APP Secret、固定共享密钥或真实测试令牌；
* 日志中的公式、文件路径、HTTP 响应和异常参数默认按可能敏感处理；
* 依赖升级、打包和发布前执行许可证、漏洞与敏感信息检查。

---

## 8. 技术架构

本节只保留产品级技术约束。规范性数据模型、线程、解析、渲染、状态和生命周期以 `docs/architecture.md` 为单一事实来源；方案理由以 `docs/decisions.md` 为准。

总体结构：

```text
UI ↔ AppController
       ├─ 所有绘图场景 → RenderActor → Engine
       └─ OCR / 截图任务 → RecognitionWorker / Service
```

Actor/Worker 是任务执行边界，不是 Engine 或 Service 的下游业务层。AppController 统一提交完整场景请求，UI 不直接启动 Actor/Worker。

### 8.1 UI 层

负责：

* 创建和显示控件；
* 接收用户输入；
* 发出用户操作信号；
* 显示状态、错误和结果；
* 将 PNG 数据转换为 Qt 显示对象；
* 根据控制器状态启用或禁用按钮。

UI 不负责：

* 直接调用 SimpleTex；
* 直接管理复杂数学解析；
* 直接构建完整 Matplotlib 绘图流程；
* 保存 API 密钥；
* 在按钮回调中实现完整业务链路；
* 判断旧任务结果是否有效。

---

### 8.2 AppController

项目从初始架构开始加入轻量 `AppController`。

它负责：

* 接收 UI 发来的用户意图；
* 创建带共享视口和 `items[]` 的 PlotSceneRequest，以及 RecognitionRequest；
* 分配任务编号；
* 保存当前有效任务编号；
* 向 RenderActor 或 RecognitionWorker 提交请求；
* 持有后台执行器的生命周期引用；
* 接收任务成功、失败和结束信号；
* 丢弃已经过期的任务结果；
* 控制 TaskPhase 与持久结果状态；
* 将可显示结果通知 UI；
* 处理应用关闭流程；
* 防止同一任务重复提交；
* 协调截图、OCR、公式确认、绘图和复制流程。

它不负责：

* 创建具体 QWidget；
* 实现 SymPy 解析细节；
* 实现 HTTP 请求细节；
* 绘制 Matplotlib 图像；
* 保存真实密钥内容。

核心 MVP 不引入依赖注入框架、事件总线或复杂命令系统。

---

### 8.3 Engine 层

Engine 负责纯数学处理：

* 输入规范化与 SourceMap；
* 逐项识别显函数、一般直线或受限圆锥曲线方程；
* 逐项安全解析等号两侧并路由到对应管线；
* 生成包含带类型函数/几何 Spec items 的 `PlotSceneSpec`；
* 验证变量、函数和结构；
* 在复杂度验证后对一次/二次方程进行受限系数提取、分类、退化和实图形验证；
* 构造显函数数值函数或直线/圆锥曲线几何参数；
* 执行分类型、参数化且受资源预算约束的采样；
* 处理定义域、NaN、无穷大和不连续点；
* 在一个 Matplotlib Figure 中生成单项或多项场景；
* 输出 PNG bytes；
* 返回警告和绘图元数据。

Engine 必须：

* 不依赖 QWidget；
* 不访问网络；
* 不操作 QClipboard；
* 不弹出窗口；
* 不读取真实配置文件；
* 输入确定时产生可预测结果；
* 可由 pytest 直接测试。

### 8.5 Actor 与 Workers 层

项目使用一个常驻 `RenderActor` 处理所有完整绘图请求，另以独立 `RecognitionWorker` 处理 OCR 网络任务。两者线程、请求编号和生命周期分离。

RenderActor：

* 长期存在、单线程、串行消费；
* 最多保留一个运行任务和一个 latest-wins 待任务；
* 接收不可变 PlotSceneRequest，由 Engine 完成解析到渲染的完整链路；
* 通过协作取消、request_id 和 scene_revision 使旧工作失效；
* 使用 Figure、FigureCanvasAgg 和 Agg，不依赖 pyplot 全局状态；
* 不返回 QWidget、QPixmap 或 QClipboard；
* 应用关闭时拒绝新任务、清除待任务、作废当前结果并有上限地等待清理；
* 不使用 `QThread.terminate()`。

RecognitionWorker 只承载 OCR 调用，不与 RenderActor 共用线程或生命周期。所有跨线程结果使用明确对象，不传无结构 dict。

推荐信号：

```text
started(request_context)
succeeded(typed_result)
failed(ErrorInfo)
finished
progress(typed_progress)  # 仅未来需要时增加
```

具体 QThread 连接和清理方式必须根据当前 Qt 官方资料确认。

## 10. 公式处理管线

推荐流程：

```text
用户确认后的 PlotSceneRequest
↓
request_validator（仅处理无需分类即可判断的请求级限制）
↓
对每个 item：normalizer / tokenizer / equation_splitter / restricted parser / plot_classifier / typed validator
↓
全部 item 验证成功后形成 PlotSceneSpec
↓
解析 ResolvedViewport 并构造 RenderPlan
↓
budget_validator
↓
typed samplers
↓
FigureCanvasAgg renderer 在共享视口和 Figure 中叠加全部 item
```

### 10.1 normalizer

负责：

* 去除外围空格；
* 统一常见全角符号；
* 处理部分 Unicode 数学符号；
* 去除明确允许移除的外围 LaTeX 标记；
* 统一当前里程碑明确支持的写法；
* 保留足够的源位置信息，以便错误提示指出问题片段。

它返回规范化文本与 SourceMap，但不是完整安全边界，不负责信任输入。

### 10.2 equation_splitter

负责：

* 区分单独表达式与含一个等号的方程；
* 单独表达式按显函数右侧候选处理；
* 将等号两侧作为两个独立表达式交给 parser；
* 拒绝零个表达式、多个等号、不等式、方程组和参数方程；
* 不在字符串阶段尝试代数求解或判断是否为圆锥曲线。

### 10.3 parser

负责：

* 先由受限 tokenizer 识别白名单 token；
* 优先构造项目自有受限 AST，并在库级代数求值前完成复杂度检查；
* 不直接信任 OCR 输出；
* 要求完整消费输入，禁止只解析合法前缀而忽略尾部；
* 返回解析树或带错误位置、用户消息和技术代码的结构化错误；
* parser 成功不代表表达式允许执行或绘图。

### 10.4 plot_classifier

负责：

* `y = rhs(x)` 且右侧不含 `y` 时路由为 `EXPLICIT_FUNCTION`；
* `lhs(x) = y` 且左侧不含 `y` 时等价交换两侧，路由为 `EXPLICIT_FUNCTION`；
* 上述特例只处理等号一侧恰为单独符号 `y` 的直接互换，不做移项、化简或通用代数求解；
* 单独表达式路由为 `EXPLICIT_FUNCTION` 候选；
* 其他单等式构造 `lhs - rhs = 0`；总次数为 1 的非退化方程路由为 `LINE_EQUATION`，总次数为 2 的方程路由为 `CONIC_EQUATION` 候选；
* 不调用通用 `solve()` 把隐式方程拆为多个显函数；
* 不因为表达式含 `x`、`y` 就全局放宽显函数白名单；
* 返回带类型的 PlotItemSpec 或明确的“不支持类型”错误。

### 10.5 validator

所有输入共同检查：

* 不允许未知自由变量；
* 只允许白名单函数；
* 只允许白名单运算；
* 限制表达式长度；
* 限制表达式树规模；
* 限制嵌套深度；
* 限制极端幂指数；
* 拒绝矩阵、积分、求和、导数等当前路线未支持结构；
* 拒绝可能造成异常耗时或内存占用的表达式。

显函数额外要求：

* 自由变量只能为 `x`；
* 不允许右侧出现 `y`；
* 必须能构造受限的一维数值函数。

圆与圆锥曲线额外要求：

* 自由变量只能为 `x`、`y`；
* 直线/圆锥曲线不允许函数或变量分母，并只接受总次数 1/2 的对应结构；
* 系数必须是有限实数；
* M1.5 不允许 `xy` 项；
* 必须通过非退化和存在实数图形验证。

#### 10.5.1 RenderPlan 与 budget_validator

负责：

* 在所有 item 分类、分类型验证且视口解析为 ResolvedViewport 后构造 RenderPlan；
* 验证 `items[]` 非空且不超过配置的项目数上限，并保证 `item_id` 唯一、顺序明确；
* 为每项确定稳定样式、采样策略和分支计划；
* 汇总单项/场景总采样点、分支数、预计内存、输出尺寸和渲染预算；
* 预算通过后才允许进入采样；
* 将 item 错误映射到稳定 `item_id` 和输入行；
* 决定场景默认坐标比例：含圆或圆锥曲线时为 equal，否则为 auto，用户明确设置优先。

### 10.6 explicit_sampler

负责：

* 在有限区间生成 x 值；
* 将 SymPy 表达式转换为 NumPy 数值函数；
* 捕获数值警告；
* 处理 NaN 和无穷大；
* 标记定义域外点；
* 对不连续点进行断线处理；
* 避免将渐近线两侧直接连接；
* 返回绘图数据和警告。

### 10.7 line/conic samplers

负责：

* 直线与有限 ResolvedViewport 矩形求交；
* 圆和椭圆按角参数采样；
* 双曲线按可见参数区间分别生成两支；
* 抛物线按开口方向和可见区间参数化；
* 捕获数值警告并拒绝非有限、异常形状或预算外结果；
* 记录采样点、分支、耗时、裁切与精度警告；
* 在完成、取消或失败后及时释放数组。

### 10.8 renderer

负责：

* 为每个场景独立创建一个 Figure；
* 使用 Figure 与 FigureCanvasAgg；
* 不调用 `pyplot.show()`；
* 绘制 typed samplers 返回的分支/线段；
* 按 items 顺序叠加全部曲线，并应用稳定的颜色、线型和图例；
* 只绘制一套共享坐标轴、网格和视口；
* 按 RenderPlan 应用 ResolvedViewport 与自动/等比例坐标；
* 检查每项是否存在实际可见分支，不把空结果当作成功曲线；
* 写入 `BytesIO`；
* 返回 PNG bytes；
* 关闭或释放对应 Figure；
* 返回尺寸、耗时、场景及逐项可见路径数量和警告信息；
* 所有 Matplotlib 操作在受控的串行边界内完成。

## 11. 公式安全策略

项目主要面向高中数学教师，因此各绘图类型采用独立受限白名单，不追求解析全部数学表达式。

### 11.1 允许内容

显函数初步允许：

```text
变量：
x

运算：
+ - * / **

常量：
pi
E

函数：
sin
cos
tan
sqrt
abs
exp
log
ln
lg
```

圆与圆锥曲线方程只允许：

```text
变量：
x
y

关系：
恰好一个等号

运算：
+ - * / **

系数：
整数、有限小数、分数
```

圆与圆锥曲线管线中的除法分母不得包含变量，幂只允许形成总次数不超过 2 的多项式。白名单的任何扩展都必须先更新 `docs/supported-formulas.md` 和参数化测试。

### 11.2 明确拒绝

所有类型共同拒绝：

* 未知变量；
* 任意函数名；
* 矩阵；
* 积分；
* 求和；
* 极限；
* 导数；
* Lambda；
* 方程求解命令；
* 任意代码结构；
* 过长输入；
* 过深嵌套；
* 极端指数；
* 超大图片；
* 超高 DPI；
* 超大采样量、分支数或输出资源；
* 无法完整消费的输入；
* 能触发属性访问、导入、调用任意对象或执行代码的结构。

此外：

* 显函数拒绝 `x` 之外的自由变量；
* 圆与圆锥曲线只额外允许 `y`，不允许其他参数；
* M1.5 拒绝含 `xy` 项、退化曲线、无实点曲线和非二次隐式方程。

### 11.3 安全原则

* 不将 OCR 结果视为可信输入；
* 禁止把用户文本交给 `eval`、`exec`、无约束 `sympify` 或等价执行逻辑；
* 解析成功不等于允许绘图；
* 解析器必须完整消费输入，不得静默忽略非法尾部；
* 所有表达式必须经过结构验证；
* 每种绘图类型分别验证允许符号和结构，不建立全局宽松白名单；
* 验证失败时返回用户可理解的提示；
* 技术细节写入日志，但不暴露敏感内容。

## 12. 数据模型的产品语义摘要

本节只说明用户可观察的模型语义，不维护 dataclass 字段清单。不可变类型、字段、联合类型、所有权和结果诊断的规范性定义统一见 `docs/architecture.md` §5、§9 和 §11；若产品行为需要新增字段，应先更新 architecture，再在这里同步用户影响。

### 12.1 输入、视口与核查语义

* `ErrorInfo` 必须提供稳定错误码、可直接显示的中文消息和可定位到字段/原文位置的信息；脱敏技术细节不得直接展示给普通用户。
* `ViewportRequest` 表达用户的 auto/manual 意图，并独占 `aspect_request`；`ResolvedViewport` 表达 Engine 得到的有限最终范围和实际比例。比例与范围都是全场景共享设置，任一变化都立即递增 revision。
* OCR 供应商结果与本地安全分析保持分离，核查界面组合展示原图、OCR 文本、规范化表达式和确定的绘图类型；供应商结果不能绕过本地验证。

### 12.2 Scene、item 与计划语义

* M0 起只有 Scene 入口；单图是只含一个 item 的场景，多图是含多个 item 的同一模型，不建立平行单图契约。
* 每个 item 在编辑期间有稳定标识，错误、警告、样式和可见性信息必须回到对应输入行。
* 全部 items 共享一套 Viewport、坐标轴、网格、图片尺寸和坐标比例；M1.6 不允许单项覆盖场景比例。
* 分类成功后的 item 必须具有确定绘图类型和对应的分类型 Spec，不能用散落布尔值模拟不同数学类型。
* 单项 M1 已在所有 item 验证和视口解析后构造 RenderPlan 并通过预算，M1.6 只扩展场景总预算、样式和逐项诊断；正式采样或渲染不得绕过计划与预算门禁。

### 12.3 结果与状态语义

* 场景结果必须携带内部请求编号和 scene revision；二者都匹配时才可成为当前结果。
* M1.6 使用原子结果：任一 item 失败、完全不可见或超限，本次场景失败且不替换上一张成功图片。
* TaskPhase 使用 `docs/architecture.md` §9 定义的六值封闭集合，只表达唯一的用户可见前台活动；首版不同时调度 OCR/截图与绘图。
* 结果存在性、复制可用、stale 和 ready 全部从核心状态派生。错误是通知而不是 TaskPhase；失败、取消或过期不能删除上一张成功图片。

## 13. 状态转换

手动绘图：

```text
IDLE → RENDERING
RENDERING 成功且 request_id/revision 均匹配 → IDLE + 更新 last_successful_result 和 last_result_scene_revision
RENDERING 失败/取消/过期 → IDLE + 保留 last_successful_result + 显示必要通知
```

加入 OCR 后：

```text
IDLE → RECOGNIZING
RECOGNIZING → REVIEWING
RECOGNIZING 失败/取消 → IDLE + 保留旧 PlotSceneResult
REVIEWING → RENDERING
REVIEWING 返回/取消 → IDLE + 保留旧 PlotSceneResult
```

加入截图后：

```text
IDLE → CAPTURING
CAPTURING → RECOGNIZING
CAPTURING 取消/超时/失败 → IDLE + 保留旧 PlotSceneResult
```

输入和结果 revision：

```text
用户编辑任何影响结果的字段 → current_scene_revision 立即递增
生成成功且 request_id/revision 均匹配 → 记录 last_successful_result 与 last_result_scene_revision
生成失败/取消/过期 → last_successful_result 不变
清空输入 → revision 递增，但不自动删除旧图
用户明确清空图片 → last_successful_result=None
```

关闭程序：

```text
任意状态 → SHUTTING_DOWN
```

处于 `SHUTTING_DOWN` 时不得启动新任务。

状态不变量：

* 复制可用性只由是否存在 last_successful_result 派生，不由最近一次任务是否失败决定；
* TaskPhase 为 `IDLE` 不代表没有结果，也不代表没有错误通知；
* 任何失败、取消或过期结果都不得覆盖当前有效 PlotSceneResult；
* REVIEWING 必须提供确认、返回修改和取消路径；
* 关闭时进行中的任务必须停止接收新工作，并在有上限的等待后完成资源清理或让结果失效。

## 14. 请求编号和过期结果

每次 OCR 和绘图任务必须由 AppController 获得进程内唯一 `request_id`。每次影响结果的编辑必须立即递增 `scene_revision`。供应商返回的 request ID 单独保存为 `provider_request_id`，不得替代内部编号。

AppController 保存：

```text
current_recognition_request_id
current_render_request_id
```

收到结果后：

* request_id 与 scene_revision 均一致：允许接纳为当前结果；
* 任一不一致：视为过期结果，不更新当前结果；
* 过期任务仍应完成资源清理；
* 日志记录任务已被忽略，但不作为用户错误显示；
* 用户取消时立即使对应编号失效，即使底层调用暂时无法中断；
* OCR 核查确认后创建新的绘图 request_id，并保留来源 recognition request_id 供追踪。

这用于避免：

```text
旧任务较晚完成
→ 覆盖新任务结果
```

`request_id` 解决任务先后；`scene_revision` 证明结果属于哪一版场景。输入文本、item 增删/重排、视口（含 `aspect_request`）、图片尺寸、网格和其他影响结果的显示配置变化均立即递增 revision，不得防抖。

实时语法提示、规范化预览等可选派生分析可使用集中配置的单次 QTimer 防抖；正式生成只由用户明确点击/提交触发。快速连续生成由 request_id 失效、revision 校验和 RenderActor latest-wins 共同处理。截图剪贴板事件使用经 Windows 原型校准的独立去抖窗口。

## 15. 截图和剪贴板防重复

Windows 当前截图协议对“已打包并具有应用身份”和“未打包 Win32 应用”的回传能力不同。M4 必须先通过 `联网确认.md` 所列官方资料和最小原型选定策略，不预先承诺截图结果一定进入剪贴板。

优先策略：

1. **已打包回调模式**：若最终采用可获得应用身份的打包方式，使用系统截图协议、相关 ID 和明确成功/取消状态接收结果；
2. **用户主动粘贴模式**：无法可靠接收系统回调时，引导用户使用系统快捷键截图，再点击“从剪贴板识别”；
3. **受限剪贴板观察模式**：只有原型证明可靠时，才在用户主动发起后的短时间窗内监听图片变化；不得常驻监听并自动上传。

若采用剪贴板观察模式，AppController 或截图协调逻辑至少维护：

* `capture_pending`；
* `last_external_image_hash`；
* `last_internal_image_hash`；
* 当前截图任务编号；
* 截图启动时间；
* 截图超时时间；
* 信号去抖窗口；
* 发起截图前的剪贴板图片基线。

内部复制流程：

```text
准备写入绘图结果
→ 记录待写入图片指纹和当前 request_id
→ 写入 QClipboard
→ 收到剪贴板信号
→ 指纹、时间窗和请求上下文均判断为内部写入
→ 不启动 OCR
```

外部截图流程：

```text
capture_pending 为真
→ 收到新的图片
→ 去抖并与启动前基线、内部图片指纹比较
→ 确认为本次用户操作产生的新截图
→ 清除 capture_pending
→ 显示待上传预览和隐私提示
→ 用户确认后启动 OCR
```

任何实现都不得只靠一次 `dataChanged`、一个 `ignore_next_clipboard_change` 布尔值或“剪贴板中有图片”就启动 OCR。截图取消、超时、外部程序同时改写剪贴板和相同图片重复截图均必须进入原型与人工测试。

## 20. 测试要求

### 20.1 Engine 自动测试

显函数必须覆盖：

* `y=x`；
* `x=y`：按左右直接互换处理，支持为显函数；
* `y+1=x+2`：M1 不做显函数移项化简，因一次方程能力尚未交付而明确拒绝；M1.5 起构造 `F(x,y)=0` 并分类为 `LINE_EQUATION` 后成功绘制。该分期期望不是永久拒绝契约；
* `y=x²`；
* `y=x^2`；
* `y=x**2`；
* `sin(x)`；
* `1/x`；
* `sqrt(x)`；
* 多余空格；
* 常见 Unicode 数学符号；
* 多个等号；
* 未知变量；
* 多变量；
* 未知函数；
* 超长输入；
* 极端指数；
* x 轴区间错误；
* NaN；
* 无穷大；
* 不连续点；
* PNG 非空；
* PNG 尺寸正确；
* 绘图警告正确返回；
* 显函数自动 y 范围得到有限合法的 `ResolvedViewport`；
* 自动范围无法可靠推导时回退到 `x,y ∈ [-10,10]` 并给出可理解提示；
* 手动视口与显式 `aspect_request` 优先，重新绘图时不被自动值覆盖。

直线、圆与圆锥曲线必须覆盖：

* `x=2`、`x+y=1`、`2x-y+3=0`、`y+1=x+2`：M1.5 一般直线和竖直直线正确，且最后一项不得走显函数通用移项旁路；
* `x^2+y^2=25`：圆且单位比例正确；
* `(x-2)^2+(y+1)^2=9`：平移圆正确；
* `x^2/9+y^2/4=1`：完整椭圆，长短轴方向正确；
* `(x-1)^2/9+(y+2)^2/4=1`：平移正确；
* `x^2/9-y^2/4=1` 与 `y^2/4-x^2/9=1`：两支存在且不误连；
* `y^2=8*x`、`x^2=-4*y`、`(y-1)^2=4*(x+2)`：开口和顶点平移正确；
* Unicode 上标、隐式乘法、空格、左右交换、整体乘非零常数；
* 分类结果和规范化系数正确；
* 参数化采样点代回规范化方程后残差在明确容差内；
* 曲线完全可见、部分裁切和完全不在视口三种情况；
* `x^2-y^2=0` 等退化曲线明确拒绝；
* `x^2+y^2=-1` 等无实点曲线明确拒绝；
* `x*y=1` 和含 `xy` 项的旋转曲线在 M1.5 明确拒绝；
* 三次方程、任意隐函数、变量分母、未知参数和零方程明确拒绝；
* 过大范围、极小跨度、巨大系数和超密采样不会崩溃或越过资源上限；
* 原有显函数用例除已明确分期升级为 M1.5 直线的 `y+1=x+2` 外全部回归通过；该用例分别保持“M1 拒绝、M1.5 成功”的阶段契约。

同一坐标系多图像必须覆盖：

* 第 4.3 节六组高中教学示例均生成一张包含全部预期曲线的图片；
* 两个及以上显函数、两个及以上圆锥曲线、显函数与圆锥曲线混合三类场景；
* 所有 items 使用同一 Viewport、坐标轴、网格、图片尺寸和坐标比例；
* 场景含圆或圆锥曲线时默认单位等比例，圆不得变形；
* 图例顺序与输入顺序一致，重新生成及增删其他项后既有 `item_id` 的颜色和线型稳定；
* 灰度截图中仍可通过线型或图例区分各项；
* 任一项存在语法、类型、资源或空视口错误时，场景整体失败，错误精确关联对应 `item_id` 和输入行，上一张成功图片不被覆盖；
* 不允许因单项采样或渲染失败而静默漏画；
* 从 1 项到配置项目数上限均可按资源预算生成，超过上限一项时被明确拒绝；场景总采样点、分支、内存和输出尺寸上限分别进行边界测试；具体上限以阶段 5 创建并在 M1.6 前校准的 `docs/supported-formulas.md` 为准；
* 单项场景与 M1、M1.5 原有结果无功能回归。

安全与健壮性必须覆盖：

* 属性访问、`__import__`、eval/exec 字符串、Lambda、代码片段、非法尾部、极端指数、超深嵌套和超长数字无法执行；
* 解析器不静默接受合法前缀并忽略非法尾部；
* 使用固定随机种子的无效字符串和结构化边界输入进行“不崩溃、不越界”测试；
* 失败后资源释放且不会留下未关闭 Figure。

不要要求不同环境生成的 PNG 二进制完全一致。

## 22. 发布验收标准

发布候选版本必须：

* 在干净 Windows 11 64 位设备上无需预装 Python 即可安装、启动、升级和卸载；
* Windows 10 1809 及以上 64 位兼容性按最终决策和实机矩阵通过，并明确其系统已结束常规支持的风险；
* 资源文件不依赖开发机绝对路径；
* 不包含真实 API 密钥、共享核心凭据或供应商 APP Secret；
* 不包含测试截图；
* 不包含内部调试文件；
* 第 4.1、4.2 节声明支持的函数和圆/圆锥曲线全部通过自动验收；
* 若 M1.6 未形成经批准的延期决定，第 4.3 节多图场景、逐项错误定位、稳定图例和资源上限全部通过验收；
* 公式、方程、图片或网络错误不会使程序退出；
* 无网络时手动绘图、预览和复制完整可用；
* OCR 超时不会冻结界面；
* 关闭程序不会遗留运行线程；
* 失败、取消和过期任务不会覆盖或误标上一张成功图像；
* 图片通过已确认版本的 PowerPoint、Word、WPS、希沃白板或替代软件粘贴矩阵；
* 在 100% 至 200% 目标缩放和多显示器场景中可用；
* 包含第三方许可证、字体和图标授权说明；
* 有经评审的隐私说明、OCR 供应商说明和本地数据清理方式；
* 有版本号、日志位置、日志清理和诊断导出说明；
* 通过依赖漏洞、敏感信息、杀毒软件误报和代码签名发布检查；
* `联网确认.md` 中所有发布阻塞项已关闭，或有负责人批准的明确降级方案。

## 23. 外部确认与决策管理

所有需要联网、供应商、法律、目标设备或最小原型确认的事项统一维护在根目录 `联网确认.md`，PRD 不再复制一份容易过期的清单。

执行规则：

* 每项记录状态、优先级、阻塞里程碑、证据链接、核实日期、结论和剩余风险；
* `已确认` 只表示资料层面的事实已找到，不等于产品方案自动获批；
* 依赖供应商行为、线程边界、截图回调、OCR 兼容率和剪贴板格式的事项必须有最小原型或实机测试；
* P0 项必须在开发步骤清单和 `联网确认.md` 指定的关闭阶段验收前关闭，或在该关闭阶段形成经项目所有者批准的明确降级决定；所有对应 P0 均关闭或降级后，相关里程碑才可完成验收，不以“里程碑开始前”制造自指门禁；
* 课程标准、Windows 生命周期、SimpleTex API 和依赖版本在发布候选前再次复查；
* 决策结论同步写入 `docs/decisions.md`，实现细节同步写入 `docs/architecture.md`。
